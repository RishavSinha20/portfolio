Case Study

Why ?

I started GoBalancer because I wanted to stop treating load balancing as a black box. In most tutorials, everything works perfectly and every backend stays up, but that is not how real systems behave. I wanted to understand what actually happens when a server slows down, crashes, or starts timing out while users are still sending traffic. Building this project from scratch helped me see the practical decisions behind routing strategy, retries, caching, and rate limiting instead of just reading about them. The goal was simple: build something close enough to a production mindset that I could reason about tradeoffs, failure behavior, and recovery with confidence.

Problem Statement

A single backend service is a bottleneck and a single point of failure. The goal was to build a traffic router that can spread load, detect unhealthy nodes, recover automatically, and fail gracefully when no healthy backend exists.

Primary Goal

Route requests reliably across multiple backend servers using strategy driven selection.

Reliability Goal

Maintain availability under partial failures via health checks, retries, and failover.

Performance Goal

Reduce backend pressure with caching and control request bursts with rate limiting.

Observability Goal

Track request and failure counts to validate if routing behavior matches expectations.

Design Decisions And Why

Reverse Proxy Core

Built on Go's reverse proxy primitives to keep request forwarding standards compliant while leaving room for custom retry and cache logic.

Multiple Routing Strategies

Implemented Round Robin, Least Connections, and IP Hash because no single policy is ideal for all workloads. Strategy is configuration driven.

Active Health Checks

Backend probes run on an interval so unhealthy targets are removed quickly, then reintroduced automatically after recovery.

Retry Before Failing

Requests get retry attempts across different targets before returning 503, improving perceived reliability during transient backend failures.

In-Memory GET Caching

Added TTL based caching for idempotent GET requests to reduce repeated backend work and improve response latency on hot paths.

Rate Limiting Middleware

Token based limiting protects downstream backends from burst traffic and makes overload behavior explicit with 429 responses.

Implementation Highlights

Config Layer

YAML controls port, strategy, and backend pool so behavior can change without code edits.

Routing Layer

Round Robin, Least Connections, and IP Hash are selected through a strategy driven balancer.

Health Layer

Periodic probes mark nodes up or down for automatic failover and recovery.

Proxy Layer

Reverse proxy forwarding applies retries and fallback 503 handling during outages.

Performance Layer

GET response caching and token-based rate limiting protect backend resources.

Metrics Layer

Atomic request and failure counters expose live system behavior for visibility.

Tradeoffs

Caching

Improves latency and reduces backend load, but introduces stale data risk and memory overhead.

Retry Logic

Increases resiliency for transient issues, but can generate extra traffic during outages.

Least Connections

Better for uneven workloads, but requires active connection accounting per backend.

Rate Limiting

Protects infrastructure under bursts, but can reject legitimate spikes with 429 responses.
