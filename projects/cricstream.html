Case Study

Why ?

I started CricStream to understand how real-time event-driven systems work at scale. Most tutorials show batch processing, but production systems need to react to events as they happen. What does a live leaderboard feed do when 100 million users watch a cricket match? How do you compute analytics without blocking the event stream? Building CricStream taught me that event-driven architecture isn't just about moving data—it's about decoupling producers from consumers, scaling processing independently, and serving real-time insights from cached state. Every piece had a purpose. The goal was practical: build something that could stream millions of cricket events, process them concurrently, compute stats in real-time, and display them live.

The Real Problem

Live sports analytics need to handle:

Massive Event Volume

Every ball in a cricket match generates multiple events (run, boundary, wicket). High volume requires a system that doesn't block on each event.

Real-Time Computation

Viewers demand instant updates: score, run rate, boundaries. Computation must happen concurrently with event ingestion.

Decoupled Producers & Consumers

Event generator shouldn't block waiting for analytics. A message queue decouples them so either can fail independently.

Live Dashboard Performance

Serving analytics directly from computation is slow. Caching results in Redis ensures dashboard gets instant responses.

Event Types Streamed

Bounded Worker Pool: The Concurrency Pattern

Instead of creating a goroutine per event (which would overwhelm the system), CricStream uses a fixed pool of workers. Events are distributed to workers via channels:

Why bounded? Unbounded goroutines lead to memory exhaustion. A pool of 8 workers processes all events with controlled resource usage. Events are queued if all workers are busy.

Why Kafka? Why Redis?

Kafka Decouples

Simulator publishes events without waiting. Consumer processes at its own pace. If consumer crashes, events are replayed from Kafka. Neither depends on the other.

Worker Pool Scales

Fixed workers ensure predictable CPU usage. Bounded queues prevent memory exhaustion. Easy to tune concurrency without changing code.

Redis Caches Results

Dashboard queries Redis, not the computation engine. O(1) lookups instead of recomputing on every request. Millions of viewers, same latency.

REST API is Stateless

API instances are replaceable. They read from Redis and serve requests. Scale by adding more API instances without state coordination.

Scalability Insights

Horizontal Kafka Scaling

Add Kafka partitions. Multiple consumer instances process partitions in parallel. Throughput scales linearly with partitions (and instances).

Worker Pool Tuning

More workers = more concurrency but more CPU. Fewer workers = less CPU but more queuing. Profile to find optimal worker count.

Redis as Bottleneck

All analytics writes go to Redis. With millions of events, Redis can bottleneck. Solution: batch updates, use Redis pipelining, add Redis replicas for reads.

Dashboard as Consumer Too

Dashboard could subscribe to a separate Kafka topic for events instead of polling API. Enables real push notifications instead of polling latency.

Lessons From Building CricStream

Event-Driven is About Decoupling

The real power isn't the message broker. It's that components don't know about each other. Simulator doesn't care about analytics. API doesn't care about Kafka.

Concurrency Without Unbounded Goroutines

Spawning a goroutine per event seems simple but crashes in production. Bounded worker pools are the production pattern for handling event streams safely.

Caching is Architecture

Redis wasn't an afterthought. It's fundamental to the design. Without it, dashboard queries would hammer the computation engine and latency would spike.

Measure Everything

Event processing, queue depths, Redis latency, API response times. Without metrics, bottlenecks are invisible. With them, scaling becomes a tuning exercise.

Technology Stack

Go Backend

Goroutines for lightweight concurrency. Channels for safe communication. Fast startup, low memory footprint.

Apache Kafka

Durable event log. Consumer groups for parallel processing. Replayable streams for debugging and recovery.

Redis

In-memory data store. Sub-millisecond response times. Perfect for real-time analytics serving.

Gin REST API

Lightweight HTTP framework. Fast JSON serialization. Easy to add endpoints for new analytics.

What This Taught Me

📚 About Event-Driven Systems: Real-world systems decouple producers and consumers through message queues. This enables independent scaling, failure isolation, and replaying history.
🔄 About Concurrency in Go: Goroutines are cheap, but creating one per event is wasteful. Worker pools with channels are the standard pattern for bounded concurrency.
⚡ About Real-Time Systems: Speed isn't just code optimization. It's architecture: decouple computation from serving, cache results, serve from cache, update in background.
📊 About Analytics at Scale: Computing all metrics on every query doesn't scale. Streaming aggregation (computing as events arrive) keeps metrics fresh and queries fast.
🏗️ About System Design: Each component (simulator, Kafka, workers, Redis, API) solves exactly one problem. Complexity comes from their interaction, not individual pieces.
