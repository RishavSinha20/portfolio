Case Study

Why ?

I started KVault to understand how systems like Redis, etcd, Consul, and ZooKeeper work internally. These systems power real infrastructure, but most tutorials gloss over the hard parts: how do you persist data safely? What happens when a node crashes? How do replicas stay in sync? Building KVault from scratch forced me to confront these questions. I evolved it from a simple in-memory store into a fault-tolerant distributed system with Write-Ahead Logging, replication, and automatic leader election. Each layer revealed new tradeoffs between durability, consistency, and performance. The goal was educational: understand the fundamental building blocks that make modern data systems reliable, so I could reason about their design with confidence.

The Evolution Path

KVault was not built all at once. It evolved in layers, each solving a new failure mode:

Phase 1: In-Memory Store

Start with a simple concurrent key-value map using RWMutex. Fast, but loses all data on crash.

Phase 2: Write-Ahead Log

Add persistence layer. Before modifying memory, write to WAL on disk. Now data survives crashes and can be replayed on restart.

Phase 3: Leader-Follower Replication

Add multiple nodes. Leader handles all writes and replicates them to followers. Now a single node failure doesn't lose data.

Phase 4: Automatic Failover

Add heartbeat monitoring and leader election. When leader dies, followers detect it and promote one to leader automatically.

Core Problem Solved

Traditional single-node key-value stores face two critical failures:

Data Loss on Crash

Without persistence, restarting the process wipes all data. In-memory only works for caches, not databases.

Unavailability on Node Failure

When the single node dies, the entire system is down. Clients lose read and write access until manual recovery.

KVault Solution: WAL + Replication

Writes survive crashes via WAL. Replicas survive node death via distributed state. Leader election restores availability automatically.

Result

A fault-tolerant key-value store that can survive any single node failure without data loss or manual intervention.

Three Core Layers

KVault's architecture separates concerns into three independent layers:

Storage Layer

In-memory concurrent hash map protected by RWMutex. Allows multiple readers, serializes writers for correctness.

Durability Layer

Write-Ahead Log (WAL) persists every operation to disk before modifying memory. On restart, replay WAL to recover state.

Replication Layer

Leader coordinates writes across follower replicas. Each replica maintains its own WAL. Heartbeats detect failures and trigger election.

Write Path: From Client to Disk to Followers

1. Client Sends Put Request
2. Leader Writes to WAL
3. Leader Updates Memory
4. Leader Sends to Followers
5. Each Follower Writes WAL
6. Followers Acknowledge
7A. Write Committed
8A. Return Success to Client
7B. Timeout on Follower
8B. Return Error to Client

Recovery Path: From Disk Back to Memory

When a node crashes and restarts, it replays its WAL to reconstruct the entire state without losing any committed data:

Node Restart
Read WAL File
Parse Entries
Replay in Order
Rebuild Memory
Join Cluster
Recovery Complete

Result: No data loss. The recovered node has identical state to before the crash and can serve reads and writes immediately.

Failover Path: When Leader Dies

Heartbeats enable automatic detection and recovery:

Leader Alive
Sends Heartbeats
Leader Crashes
Heartbeat Timeout
Follower Becomes Candidate
Requests Votes
New Leader Elected

Once elected, the new leader sends heartbeats and the cluster continues to accept reads and writes.

Design Tradeoffs

WAL Persistence

Pros: Crash safety, recovery without data loss. Cons: Slower writes, disk I/O overhead.

RWMutex for Concurrency

Pros: Multiple readers, good for read-heavy workloads. Cons: Simple mutex would be easier, Raft-style locking is complex.

Leader-Follower Model

Pros: Simple consistency model, centralized coordination. Cons: Can't tolerate leader+quorum failures, write bottleneck at leader.

Heartbeat Frequency

Pros: Frequent heartbeats detect failures quickly. Cons: More network traffic, false positives on slow networks.

Key Insights and Lessons

Why Durability Matters

Without WAL, every crash loses data. WAL turns transient memory into durable storage by forcing writes to disk before acknowledgment.

Replication Solves Availability

A single node is a single point of failure. Replication distributes state so any one node can fail without losing data.

Heartbeats Are Simple but Powerful

Detecting failures via heartbeat timeout is simple and effective. It enables automatic recovery without complex consensus protocols.

Layer Separation is Key

By separating Storage, Durability, and Replication into independent layers, each can be understood and debugged in isolation.

What This Taught Me

Building KVault gave me deep intuition for distributed systems that I couldn't get from reading papers. I now understand:

How Redis Persistence Works

RDB snapshots and AOF logs are different ways to solve the same problem: how to make in-memory state durable.

Why Raft Exists

Leader-follower replication with heartbeats is simple but brittle. Raft adds voting and term numbers to handle edge cases safely.

The Cost of Replication

Faster writes require fewer replicas. Durability requires slower writes. Real systems tune this based on risk tolerance.

Failure Recovery is Hard

Most bugs happen during recovery. Replaying state is tricky and edge cases hide in rare failure scenarios.
